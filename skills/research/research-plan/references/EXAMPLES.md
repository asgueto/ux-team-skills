## Example 1: User Interviews + Shadowing (GL Share)

**Study:** GLShare - restrict/public user interviews  
**Product:** GL Share  
**Method:** User interviews with prototype, Shadowing  
**Researchers:** Maciej Kosilo, Nat Guerrero

### Goals (v3 — after two rounds of refinement)
1. Know whether users are able to understand the difference between "Public" and "Restricted" access types when creating a Share (as a sender)
2. Understand user preferences and habits on Public or Restricted depending on personal or professional use
3. Understand if users ("Senders") are able to understand and choose the permissions within the modal to their use case:
   - *Any recipient can forward without requesting sender approval*
   - *Only specific recipients can forward without requesting sender approval*
4. Ensure correct usability and understanding of creating a new share from received files

**Why this works:** Each goal is framed as something to *understand* or *evaluate*. Goal 3 is detailed enough to anticipate the prototype tasks. The versioning (v1 → v3) shows the team sharpened scope iteratively.

### Participants
- 5 client users from user club ($50 each)

- Previous research on Restricted/Public shares

## Example 2: Interviews + Shadowing (Internal workflow)

**Study:** GL Live/eTolk client feedback process  
**Product:** GL Live Booking / InterAct  
**Method:** User interviews, Shadowing  
**Stakeholders:** Justyn Vasquez, Angel Lopez (Semantix)

### Goals
Understand how client feedback is collected and handled in Semantix and TransPerfect.

**Goal/question table:**

| 🎯 Goal | ❓ Driving question |
|---|---|
| Understand the type of feedback clients leave | What is the feedback about, in general? |
| Identify which type is most common | What is the most common type of feedback received? |
| Identify how clients can leave feedback | What are the different ways clients can leave feedback? |
| Identify which option is most used | Which option is preferred? |
| Identify who handles feedback when it arrives | Who handles the feedback in the first instance? |
| Define the main steps of the feedback process | What are the general steps taken when feedback arrives? Are there protocols? |
| Identify which departments can be involved | Which departments can be involved in handling feedback? |
| Identify who is responsible for investigating | Who is responsible for investigating feedback? |
| Discover what actions can be taken in response | What actions can be taken in response to client feedback? |
| Identify how progress is tracked | How do you keep track of the feedback? (tickets?) |

**Why this works:** The goal/question table is especially useful for stakeholder-facing plans and for briefing note-takers. Each row maps directly to a section of the discussion guide.

### Participants
- Type: Production Teams
- Quantity: 2 primary contacts (1 TransPerfect, 1 Semantix), plus other key players identified during research

### Steps
- [x] Kick-off
- [x] Goals
- [x] Identify contacts
- [x] Conduct interviews
- [x] Analyse
- [x] Share

## Example 3: User Interviews (Internal stakeholder research)

**Study:** InterAct - Sales role  
**Product:** GL Live Booking / InterAct  
**Method:** User interviews  
**PM:** Angel Lopez (Semantix)

### Goals
1. Understand how to better integrate ProjectA with InterAct so Sales managers can use InterAct on a daily basis
2. Understand limitations of the ProjectA integration
3. Understand how to introduce the Sales Role to InterAct
4. Understand how to make communication between Client, Sales, and PMs effective in one platform
5. Evaluate TransPerfect's previous experience integrating TransPort and ProjectA
6. Understand how Sales managers will work effectively in InterAct
7. Understand advantages and disadvantages of TransQuote
8. Understand how PMs will collaborate with Sales managers in InterAct
9. Understand different kinds of Sales workflow (requests, quotes, enterprise accounts)

**Expected outcomes stated explicitly:**
> "In the end of the research we will have: Full understanding and diagram of Sales managers' different workflows; Proposal of ideal workflow for Sales with integration to accounting tool; All information about how to build an integration of InterAct with ProjectA (or TransQuote or any other tool identified)"

**Why this works:** The expected outcomes section sets explicit deliverable expectations with stakeholders before a single session is run. This prevents the "we did the research but no one used it" outcome.

### Participants
- Sales Managers (interpretation and translations)
- Project Managers (interpretations)
- TransPort Team (Customer support)
- TransQuote Team
- ProjectA team

### Steps
1. Create script (questions for different user types)
2. Conduct interviews
3. Create current workflows
4. Create ideal workflow
5. Share with stakeholders
6. Understand next steps to integration

## Goal Writing: Before / After

### Too broad
❌ "Understand the user experience of GL Share"

Better: break into specific angles
✅ "Understand when users choose Public vs. Restricted and why"
✅ "Evaluate whether users can successfully complete a forward-share without assistance"

### Too narrow / task-framed
❌ "Ask users about forwarding"
✅ "Understand how users interpret the forwarding permission options and what they expect to happen"

### Activity disguised as a goal
❌ "Conduct 5 usability tests on the new booking flow"
✅ "Identify where users lose confidence or make errors in the new booking flow"

## Participant Descriptions: Patterns

**For client-facing product studies:**
> "5 client users from user club ($50 each)"
> "35 participants from industries including Public Administration, Medical (+ER), Police"

**For internal/workflow studies:**
> "2 primary contacts with deep process knowledge (1 from TransPerfect, 1 from Semantix), plus other key players identified during research"
> "Sales Managers, PMs, TransPort Team, TransQuote Team, ProjectA team"

**For unmoderated tests (note the quantity):**
> "35 participants of different roles" — at this scale, quantitative patterns become meaningful (e.g. "86% completed task 1 correctly")

## Checklist Reference

### Minimal (quick evaluative study)
- [ ] Goals defined and aligned with PM
- [ ] Participants recruited
- [ ] Script reviewed
- [ ] Sessions conducted
- [ ] Report shared

### Standard (moderated UT or interviews)
- [ ] Kick-off with PM and stakeholders
- [ ] Goals defined (versioned if needed)
- [ ] Participants identified
- [ ] Recruitment confirmed
- [ ] Script written and reviewed
- [ ] Prototype / materials ready
- [ ] Sessions conducted
- [ ] Analysis
- [ ] Report / readout in Condens
- [ ] Findings shared

### Extended (generative / shadowing / multi-method)
- [ ] Kick-off
- [ ] Goals (v1 → iterate)
- [ ] Stakeholder alignment on scope
- [ ] Assumptions documented
- [ ] Participants identified and recruited
- [ ] Consent forms prepared
- [ ] Script / discussion guide written
- [ ] Note-taking structure set up in Condens
- [ ] Sessions conducted
- [ ] Debrief after each session
- [ ] Full analysis
- [ ] Deliverables defined (report format, audience)
- [ ] Report in Condens
- [ ] Findings shared with stakeholders
- [ ] Next steps / follow-up documented
